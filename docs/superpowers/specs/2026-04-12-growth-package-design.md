# Stacklit Growth Package — Design Spec

## Goal

Ship 5 improvements that make Stacklit more adoptable, more shareable, and more useful. Target audience: individual devs using AI coding tools, teams wanting standardized codebase docs, and OSS maintainers who want AI-friendly repos.

Success metric: move from 15 stars to 100+ through organic discovery and sharing.

---

## 1. GitHub Action (`glincker/stacklit-action`)

### What it does

A GitHub Action that keeps `stacklit.json` and `DEPENDENCIES.md` up to date automatically. Two modes:

- **auto-commit** (default): runs `stacklit generate` on push, commits changes if the index changed
- **check**: runs `stacklit generate --check`, fails CI if the index is stale

### Usage

```yaml
# .github/workflows/stacklit.yml
name: Stacklit
on: [push]
jobs:
  index:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: glincker/stacklit-action@v1
        with:
          mode: auto-commit  # or "check"
```

### Implementation

- `action.yml` at repo root of `glincker/stacklit-action`
- Downloads the latest Stacklit release binary for linux-amd64
- Runs `stacklit generate`
- In auto-commit mode: checks `git diff`, commits and pushes if changed
- In check mode: runs `stacklit diff`, exits non-zero if stale
- Inputs: `mode` (auto-commit|check), `version` (latest by default), `args` (extra flags)

### Files

- New repo: `glincker/stacklit-action`
  - `action.yml`
  - `entrypoint.sh`
  - `README.md`

### Testing

- Test on Stacklit repo itself
- Test on a sample Node.js repo
- Verify both modes work

## 2. Benchmark post (GitHub Discussion)

### What it does

A GitHub Discussion (Announcements category) that proves Stacklit saves tokens and tool calls with real numbers.

### Content structure

1. **Hook**: "We gave Claude Code a task on a 108k-line codebase. Without Stacklit: 47 tool calls, 3 minutes. With Stacklit: 8 tool calls, 30 seconds."
2. **Setup**: Express.js repo (well-known, relatable) + Stacklit repo itself
3. **Test**: Same prompt to Claude Code, measure tool calls, tokens, time, correctness
4. **Results table**: side-by-side comparison
5. **How to try it**: `npx stacklit init` one-liner
6. **Screenshots**: terminal output showing the difference

### Metrics to capture

- Number of Grep/Glob/Read tool calls before finding what's needed
- Total tokens consumed in the conversation
- Wall-clock time to answer
- Whether the answer was correct

### Where it lives

- GitHub Discussion in `glincker/stacklit` (Announcements category)
- Link from README under a "Does it actually help?" section

## 4. Shareable visual map

### What it does

Adds a "Share" button to `stacklit.html` that lets users share their codebase visual map without needing a server.

### How it works

1. User clicks "Share" in the HTML viewer
2. The current view state is serialized into a self-contained HTML string
3. Two options:
   - **Copy HTML to clipboard** — user can paste into a Gist manually
   - **Download as standalone HTML file** — user can host anywhere or share directly

The HTML is already self-contained (all data is inline). The share feature just needs to package it cleanly.

### Implementation

- Add a "Share" button to the HTML template in `assets/template.html`
- JavaScript function that:
  - Clones the current document
  - Strips any local-only elements (file paths, etc.)
  - Offers download as `.html` file
  - Offers copy-to-clipboard of the raw HTML
- Add a banner at the top of shared maps: "Generated by Stacklit — make any repo AI-agent-ready"

### Files changed

- `assets/template.html` — add Share button + JS

## 5. Framework detection

### What it does

Detects common frameworks and adds framework-specific context to `stacklit.json` so AI agents get better guidance.

### Frameworks to detect (phase 1)

| Framework | Detection signal |
|-----------|-----------------|
| Next.js | `next.config.js/ts`, `app/` or `pages/` dir |
| Express | `express` in package.json deps |
| FastAPI | `fastapi` in requirements.txt or pyproject.toml |
| Django | `django` in requirements.txt, `manage.py` |
| Spring Boot | `spring-boot-starter` in pom.xml or build.gradle |
| Gin | `github.com/gin-gonic/gin` in go.mod |
| Rails | `Gemfile` with `rails`, `config/routes.rb` |
| Axum | `axum` in Cargo.toml |

### Schema addition

```json
{
  "tech": {
    "frameworks": [
      {
        "name": "nextjs",
        "version": "14.2.0",
        "config_files": ["next.config.ts", "tailwind.config.ts"],
        "patterns": {
          "routes": "app/",
          "api": "app/api/",
          "middleware": "middleware.ts"
        }
      }
    ]
  }
}
```

### Implementation

- New file: `internal/detect/frameworks.go`
- Extend existing `internal/detect/` package which already handles language detection
- Each framework gets a detection function that checks for config files and dependency declarations
- Results populate the `tech.frameworks` field in the index
- Patterns are framework-specific and hand-coded (not derived)

### Files changed

- `internal/detect/frameworks.go` (new)
- `internal/detect/frameworks_test.go` (new)
- `internal/schema/index.go` — add `Framework` struct to schema
- `internal/engine/engine.go` — call framework detection during scan

## Implementation order

1. **GitHub Action** — highest ROI, makes adoption sticky
2. **Benchmark post** — proof of value, shareable
3. **Comparison content** — SEO capture, low effort
4. **Shareable visual map** — viral loop
5. **Framework detection** — feature depth, last because it's the most code

## Out of scope

- Hosted web service for visual maps (deferred, use gist/download for now)
- Paid tier or analytics
- Language support beyond current 11
- Auto-PR mode for the GitHub Action (just auto-commit for v1)
